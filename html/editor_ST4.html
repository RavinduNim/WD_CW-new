User Profile Page

HTML Structure

I used the shared HTML template to maintain consistency of our group website. Inside the <main> section, I created a clean structure using <div> containers for the profile output cards after geting nput fro user. These cards were initially hidden using CSS class profileCard and using CSS property display:none; . I also structured a progress bar wrapper to visually show completion with CSS progressBar class.

JavaScript Techniques

Instead of standard HTML forms, I used JavaScript prompt() clause to collect user data from browser's popup input message box progressively across three steps . I used DOM manipulation (document.getElementById) to:

  • Update the HTML content dynamically without reloading the page.
  • Display the hidden profile cards once data was filled.
  • Calculate the number of completed prompts and update the CSS width of the progress bar with completeFields++ in the for loop .
  • Allow users to skip input prompts (handling null values) and recompletee them later.

Design and Implementation Rationale

I chose a simple, global variable approach for my JavaScript to keep the logic clean and readable. By avoiding complex codes or objects, it was easier to track empty answers and completed answers for the progress bar. The progressive disclosure design keeps the user from feeling complereness by too many questions at once.

Sitemap Page (SVG)

HTML & SVG Structure

The Sitemap page was built using Scalable Vector Graphics (<svg>) to create a visual hierarchy of our website. I mapped out other main pages like Home page, Gallery page, AIS page, Feedback, Team and Profile page using shapes like <rect> and connected them with <line> tags to connect visual reqtangle via lines.

CSS & Interactive Techniques

Every node in the SVG is wrapped in an <a> tag so it functions as a working link to the actual HTML pages ad described above. I used CSS to apply hover effects (changing the fill colour) to the SVG shapes, providing clear visual feedback to the user that the diagram is interactive and user friendly.

Design and Implementation Rationale

I used SVG rather than a standard image because SVGs are resolutions independent and scale perfectly on mobile and desktop screens without losing quality and responsiveness. The design matches our group's "Life on Land" green colour palette to maintain the visual identityy of the website.

Content Page - Tech Waste and Soil Pollution

HTML Structure

My content page focuses on the environmental impact of electronic waste. I structured the page using proper website hierarchy, starting with an <h1> and using <h2> for the six subsections. Because the content is long and requires scrolling, I have implemented an internal navigation menu (List of Contents) using anchor links (href="#sec1").

The page includes a Go to Top button implemented purely with HTML and CSS, without involving on JavaScript to go to top of the page.

CSS Techniques

I used embedded CSS to style the required images, using width: 100% and max-width to ensure that were responsive, while adding borders and shadows. For the "Go to Top" button, I used position: fixed so it stays visible in the corner of the screen while scrolling.

Design and Implementation Rationale

I wrote original content focused on tech waste because it is highly relevant to IT undergraduade students while directly addressing land degradation under SDG topics. The internal anchors and fixed "Go to Top" button were added to highlt improve user experience and navigation on text heavy pages.

Challenges and Lessons Learned

My main challenge was implementing the JavaScript for the User Profile. Other one is, calculating the progress bar and handling skipped prompts was littlebit complex. I solved this by using simple if statements to check if variables were empty or null before updating the DOM and the progress percentage.

Another challenge was making the SVG sitemap accessible. I learned that standard HTML text isn't always read correctly inside SVGs, so I had to add <title> tags inside the SVG links. so screen readers could recognize the destinations properly.

From this process, I learned:

  • how to use JavaScript prompt() to get popup inputs.
  • how to manipulate the DOM to dynamically display hidden content.
  • how to code interactive graphics using SVG.
  • how to use CSS position: fixed and internal anchor links.

Accessibility Considerations

I ensured accessibility across all my assigned tasks. For the Tech Waste Content Page, I structured headings logically (H1, H2) for screen readers and included descriptive alt text for all images.

For the SVG Sitemap, I ensured all clickable nodes included a text label and an accessible name (using <title> tags). I also verified that the links are focusable via keyboard navigation. On the User Profile, I used high eye detective colours for the progress bar to ensure it is easy to read.

I also considered readability by using:

  • Use theme related colours for best visibility
  • clear headings and section structure
  • consistent text and spacing
  • responsive screen layout for diffrence device sizes

These choices improve usability and helps for a wider range of users. The brief requires accessibility to be considered across implemented pages, and the compliance notes also emphasise accessible, inclusive web design.

Compliance with JANET Regulations

In the pages I created, I adhered to JANET acceptable use policies by ensuring all published content is strictly educational and relevant to our SDG academic topic (Tech Waste). I did not include any malicious scripts or inappropriate content.

My JavaScript data collection on the User Profile is totally client side; no personal data is collect to or stored on a server, respecting user privacy. All external images used on my content page are sourced from royalty free sites and properly referenced, avoiding copyright infringement.

In addition, I considered accessibility as part of responsible publication. JANET compliance notes explain that web developers in education should use ethical content, respect copyright, avoid unnecessary data collection, and support accessibility and secure web practice.
