User Profile Page

HTML Structure

I used the shared HTML template provided by Student 1 to maintain consistency[cite: 678]. Inside the <main> section, I created a clean structure using <div> containers for the profile output cards. These cards were initially hidden using CSS [cite: 706-708]. I also structured a progress bar wrapper to visually show completion.

JavaScript Techniques

Instead of standard HTML forms, I used JavaScript prompt() dialogs to collect user data progressively across three steps [cite: 686-698]. I used DOM manipulation (document.getElementById) to:

  • Update the HTML content dynamically without reloading the page [cite: 709-712].
  • Reveal the hidden profile cards once data was entered.
  • Calculate the number of completed prompts and update the CSS width of the progress bar [cite: 713-721].
  • Allow users to skip prompts (handling null values) and revisit them later [cite: 723-738].

Design and Implementation Rationale

I chose a simple, global variable approach for my JavaScript to keep the logic clean and readable. By avoiding complex arrays or objects, it was easier to track empty versus completed answers for the progress bar. The progressive disclosure design keeps the user from feeling overwhelmed by too many questions at once.

Sitemap Page (SVG)

HTML & SVG Structure

The Sitemap page was built using Scalable Vector Graphics (<svg>) to create a visual hierarchy of our website [cite: 799-801]. I mapped out the parent and child pages using shapes like <rect> and connected them with <line> tags [cite: 802-808].

CSS & Interactive Techniques

Every node in the SVG is wrapped in an <a> tag so it functions as a working link to the actual HTML pages [cite: 814-816]. I used CSS to apply hover effects (changing the fill colour) to the SVG shapes, providing clear visual feedback to the user that the diagram is interactive [cite: 817-819].

esign and Implementation Rationale

I used SVG rather than a standard image because SVGs are resolution-independent and scale perfectly on mobile and desktop screens without losing quality [cite: 830-833]. The design matches our group's "Life on Land" green colour palette to maintain the visual identity [cite: 837-841].

Content Page - Tech Waste and Soil Pollution

HTML Structure

My content page focuses on the environmental impact of electronic waste. I structured the page using proper semantic hierarchy, starting with an <h1> and using <h2> for the six subsections [cite: 904-908]. Because the content is long and requires scrolling, I implemented an internal navigation menu (List of Contents) using anchor links (href="#sec1") [cite: 909-915]

The page includes a Go to Top button implemented purely with HTML and CSS, without relying on JavaScript [cite: 930-936].

CSS Techniques

I used embedded CSS to style the required images, using width: 100% and max-width to ensure they were responsive, while adding borders and shadows [cite: 938-946]. For the "Go to Top" button, I used position: fixed so it stays visible in the corner of the screen while scrolling[cite: 937].

Design and Implementation Rationale

I wrote original content focused on tech waste because it is highly relevant to IT students while directly addressing land degradation under SDG 15 [cite: 895-898, 902]. The internal anchors and fixed "Go to Top" button were added to vastly improve user experience and navigation on text-heavy pages [cite: 922-929].

Challenges and Lessons Learned

My main challenge was implementing the JavaScript for the User Profile. Initially, calculating the progress bar and handling skipped prompts was tricky. I solved this by using simple if statements to check if variables were empty or null before updating the DOM and the progress percentage.

Another challenge was making the SVG sitemap accessible. I learned that standard HTML text isn't always read correctly inside SVGs, so I had to add <title> tags inside the SVG links so screen readers could announce the destinations properly [cite: 823-825].

From this process, I learned:

  • how to use JavaScript prompt() to interact with users [cite: 686-687].
  • how to manipulate the DOM to dynamically display hidden content [cite: 709-712].
  • how to code interactive graphics using SVG [cite: 800-801].
  • how to use CSS position: fixed and internal anchor links [cite: 909-915, 937].

Accessibility Considerations

I ensured accessibility across all my assigned tasks. For the Tech Waste Content Page, I structured headings logically (H1, H2) for screen readers and included descriptive alt text for all images [cite: 904-908, 939-941].

For the SVG Sitemap, I ensured all clickable nodes included a text label and an accessible name (using <title> tags) [cite: 823-825]. I also verified that the links are focusable via keyboard navigation [cite: 826-828]. On the User Profile, I used high-contrast colours for the progress bar to ensure it is easy to read.

I also considered readability by using:

  • Use theme related colours for best visibility
  • clear headings and section structure
  • consistent text and spacing
  • responsive screen layout for diffrence device sizes

These choices improve usability and inclusivity for a wider range of users. The brief requires accessibility to be considered across implemented pages, and the compliance notes also emphasise accessible, inclusive web design.

Compliance with JANET Regulations

In the pages I created, I adhered to JANET acceptable use policies by ensuring all published content is strictly educational and relevant to our SDG 15 academic topic (Tech Waste) [cite: 1030-1031]. I did not include any malicious scripts or inappropriate content[cite: 1033].

My JavaScript data collection on the User Profile is entirely client-side; no personal data is transmitted to or stored on a server, respecting user privacy [cite: 1032-1035]. All external images used on my content page are sourced from royalty-free sites and properly referenced, avoiding copyright infringement[cite: 1034].

In addition, I considered accessibility as part of responsible publication. JANET compliance notes explain that web developers in education should use ethical content, respect copyright, avoid unnecessary data collection, and support accessibility and secure web practice.
